# Design: Criteria Weights, Negative Checks Page, and Documentation Hub

**Date:** 2026-03-29
**Scope:** WebStress demo site (`webagentbench/environments/demo-site/`)

---

## Overview

Three features for the demo site:

1. **Implied weight display** on evaluation criteria so the score breakdown is transparent
2. **Negative checks dashboard** at `/results/negative-checks` showing all guard-rail checks and their trigger status
3. **Full documentation hub** at `/docs` with sub-pages covering the entire LLMOS project

---

## Feature 1: Implied Weight on Criteria

### Problem

The score formula is `base_score - penalties + trajectory_mod`, where `base_score = passed_criteria / total_criteria`. Failed criteria reduce the base score by `1/n`, but this weight is never shown in the UI. Users see `✗ PA visibility correct` with no indication of how much it cost.

### Solution

Display the implied weight on failed criteria in `EvalCriteria.tsx`.

### Data flow

No changes to `prepare-results.py` or the data pipeline. The weight is computed client-side:

```
weight = 1 / count(criteria where penalty is null)
```

This separates regular criteria (which have `penalty: null` in the JSON) from negative checks (which have `penalty: <number>`).

### Visual treatment

- **Failed regular criterion:** amber badge showing `weight: 0.20` (distinct from red penalty badge)
- **Failed negative check:** existing red badge showing `penalty: -0.15` (unchanged)
- **Passed criterion:** no weight/penalty shown

### Files modified

- `src/components/ui/EvalCriteria.tsx` — add weight computation and badge rendering in both compact and full layouts

## Feature 2: Negative Checks Dashboard

### Route

`/results/negative-checks` — accessible as a sub-page of Results.

### Data source

New file generated by `prepare-results.py`: `/public/results/negative-checks.json`

#### Schema

```typescript
interface NegativeChecksData {
  total_tasks_with_negatives: number;
  total_negative_checks: number;
  triggered_count: number;
  checks: NegativeCheckEntry[];
}

interface NegativeCheckEntry {
  task_id: string;
  task_title: string;
  difficulty: string;
  desc: string;
  penalty: number;
  triggered: boolean;  // true when passed === false
}
```

#### Generation

`prepare-results.py` iterates all trajectory files. For each criterion with `penalty !== null`, it emits an entry. `triggered` is `!passed`. The summary counts are computed from the entries.

### Page layout

1. **Summary stats row** — three cards: Total negative checks, Triggered count, Trigger rate %
2. **Filter controls** — segmented toggle: All / Triggered / Clear; plus a text search on description
3. **Table** — columns: Task (links to `/results/[taskId]`), Description, Penalty, Status (green check / red X), Difficulty
4. Sorted by: triggered first, then by penalty descending

### Navigation

- Link from the Results summary page (button or text link near the top)
- Direct URL `/results/negative-checks`
- Not in the main Nav bar (it's a sub-view of Results)

### Files created

- `src/app/results/negative-checks/page.tsx` — the page component
- `src/lib/negativeChecks.ts` — data loader (`fetchNegativeChecks()`)

### Files modified

- `scripts/prepare-results.py` — emit `negative-checks.json`
- `src/app/results/page.tsx` — add link to negative checks page

## Feature 3: Documentation Hub

### Route structure

```
/docs                  — landing page (overview + cards)
/docs/scoring          — evaluation & scoring mechanics
/docs/primitives       — 12 cognitive primitives taxonomy
/docs/architecture     — system architecture
/docs/benchmark        — WebStress specifics
/docs/training         — training pipeline
```

### Shared layout

A three-column layout used by all `/docs/*` pages:

```
+------------------+------------------------+------------------+
| Left sidebar     | Content                | Right sidebar    |
| (section nav)    | (max-width ~720px)     | (page TOC)       |
+------------------+------------------------+------------------+
```

- **Left sidebar:** Persistent section links to all docs sub-pages. Active page highlighted. Collapses to a dropdown on narrow viewports.
- **Right sidebar:** Auto-generated table of contents from `h2`/`h3` headings on the current page. Scroll-spy highlights the active section. Hidden on narrow viewports.
- **Content area:** Prose, code blocks, diagrams, tables. Max-width 720px for readability. Uses existing site typography (DM Sans / DM Mono) and CSS variable theming.

### Implementation approach

All pages are static React components (`"use client"` where interactivity is needed for scroll-spy/TOC). Content is hand-authored JSX, sourced from existing project documentation but rewritten for a public audience. No markdown parsing or build-time ingestion.

Sub-pages use fixed routes (not dynamic `[param]` segments), so they are statically generated automatically by Next.js with no special configuration.

### Layout file

`src/app/docs/layout.tsx` — shared docs layout with left sidebar and right TOC slot. Each page passes its heading structure to the layout for TOC generation.

### Navigation

"Docs" added to the main Nav bar (`Nav.tsx`) between "Tasks" and "Results".

### Sub-page content outline

#### `/docs` (landing)
- Hero: "Documentation" heading with one-line description
- Three cards linking to the main subsystem docs: LLMOS Simulator, WebStress, Training Pipeline
- Quick links to all sub-pages

#### `/docs/scoring`
- Score formula with visual breakdown
- Base score: `passed / total` with implied weights
- Negative checks: guard-rail penalties
- Trajectory modifier: step efficiency bonus/malus
- Worked example using a real task (e.g., `gmail_thread_detective`)
- Link to the negative checks dashboard

#### `/docs/primitives`
- Grid of 12 primitives using the existing `PrimitivePill` component style
- Each primitive: definition, condensed research basis, which tasks test it
- Expandable detail sections

#### `/docs/architecture`
- Sandwich architecture diagram (Python + LLM separation)
- Unified agent format: indexed accessibility tree with code example
- Adapters: `llmos_adapter.py`, `playwright_adapter.py`
- State visibility rules table (simulator/agent/judge)
- Episode loop: Reset -> Act -> Step -> Evaluate
- Multi-provider LLM support (openai, gemini, vllm)

#### `/docs/benchmark`
- Task structure: YAML schema, seed system, actors, eval checks
- Gmail environment: how fixtures work, the mutator/seeder pipeline
- Version registry (condensed table from README)
- Running the benchmark: CLI commands, options

#### `/docs/training`
- Overview: SFT then DPO on Qwen models
- Data collection from simulator and WAB trajectories
- `prepare_data.py` pipeline
- Tinker API integration for cloud GPU training
- Model registry and evaluation loop
