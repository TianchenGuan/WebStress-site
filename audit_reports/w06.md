**Notes on cross-cutting findings:**

1. **React-controlled form inputs** — Throughout the patient portal, text inputs and textareas use React's controlled-component pattern. Browser automation tools that set `element.value` directly (without triggering React's synthetic events) will silently fail to register input, leaving submit buttons disabled or forms submitting empty data. This affects all tasks that require typing into reason/appeal fields. Human users typing normally are unaffected. This is a known automation-vs-React friction, not a UX bug per se, but worth documenting as a potential false-negative source in agent evals.

2. **Reset button session breakage** — The toolbar Reset button generates a new session ID client-side but the server does not know about it, resulting in 404 errors on all subsequent API calls. This makes the Reset flow completely non-functional and could trap an agent in a broken session mid-task.

3. **Appeal textarea disabled until non-empty** — The Submit Appeal button is disabled until the reason textarea has content. Since the textarea is React-controlled, agents using plain DOM value setting will see a permanently-disabled button with no error feedback. The failure mode is silent.
